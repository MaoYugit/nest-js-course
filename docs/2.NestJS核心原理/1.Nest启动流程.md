## 第一讲：创建第一个 Nest 项目

---

### 一、 环境准备与工程创建

在实际开发中，我们统一使用 NestJS 官方提供的 CLI 工具，它可以确保项目结构的标准化。

**1. 安装 CLI 命令行工具**
```bash
pnpm add -g @nestjs/cli
```

**2. 初始化项目**

```bash
nest new project-name
```
*   **开发选择**：建议选择 `pnpm` 或 `npm`。在企业级架构中，`pnpm` 因其硬链接机制能极大节省磁盘空间并加快安装速度。

---

### 二、 标准目录结构解析（核心文件）

NestJS 的工程化程度极高，初学者必须理解生成的每一个关键文件的职责：

1.  **`main.ts` (入口文件)**：使用 `NestFactory` 创建应用实例，是整个程序的启动引擎。
2.  **`app.module.ts` (根模块)**：整个应用的“地图”，负责组织和链接所有的 Controller 和 Provider。
3.  **`app.controller.ts`**：处理 HTTP 请求，属于路由层。
4.  **`app.service.ts`**：处理具体业务逻辑，属于服务层（Provider）。
5.  app.controller.spec.ts：*控制器的单元测试。*
6.  **`nest-cli.json`**：项目编译与打包的配置文件。

---

### 三、 启动与开发模式

**1. 常用命令**
*   `pnpm start`: 普通启动（不常用）。
*   `pnpm start:dev`: **热重载启动（最常用）**。底层使用的是 `tsc --watch`，监听文件变化并自动重新编译。
*   `pnpm build`: 生产环境打包。将 TS 代码编译为标准 JS 存放在 `dist` 目录。

**2. 核心依赖说明**

*   `@nestjs/core`: Nest 核心运行引擎。
*   `@nestjs/common`: 包含大量装饰器、接口和工具类（如 `@Controller`, `@Injectable`）。
*   `reflect-metadata`: 实现装饰器反射的核心库，DI（依赖注入）的基石。

---

### 四、 实际开发建议

1.  **Strict Mode**: 确保 `tsconfig.json` 中的 `strict: true` 开启。NestJS 深度依赖类型系统，关闭严格模式会丧失 TS 的大部分优势。
2.  **平台选型**: NestJS 默认底层使用 **Express**。如果追求极致吞吐量，可以在创建项目后将其切换为 **Fastify**。

---

### 五、 面试题储备

**Q1：NestJS 相比于原生 Express/Fastify 的核心价值是什么？**
*   **A**: 
    1.  **架构约束**：Express 过于灵活，导致大型项目代码凌乱。NestJS 强制使用模块化和分层架构，解决了 Node.js 项目的可维护性痛点。
    2.  **高度解耦**：内置 IoC 容器**（Inversion of Control Container，控制反转容器）**，通过 DI（**Dependency Injection，**依赖注入）实现代码解耦，便于单元测试。
    3.  **开箱即用**：集成了异常过滤器、守卫、拦截器等企业级切面（AOP）功能。

**Q2：`main.ts` 中的 `NestFactory.create(AppModule)` 背后发生了什么？**
*   **A**: 这是项目的初始化起点。它会：
    1.  初始化 **IoC 容器**。
    2.  递归扫描 `AppModule` 及其引入的其他模块。
    3.  解析模块间的依赖关系，并实例化所有相关的 Provider。
    4.  绑定 Controller 路由到具体的底层 Web 平台（如 Express）。

---

## 第二讲：main.ts 到底干了什么？

`main.ts` 是整个 NestJS 应用程序的**心脏**。虽然代码看上去只有寥寥几行，但它触发了整个框架的生命周期。

---

### 一、 标准 `main.ts` 代码拆解

```typescript
import { NestFactory } from '@nestjs/core';
import { AppModule } from './app.module.ts';

async function bootstrap() {
  // 1. 实例化阶段：通过根模块创建一个 Nest 应用实例
  const app = await NestFactory.create(AppModule);
  
  // 2. 监听阶段：启动 HTTP 服务器，监听端口
  await app.listen(3000);
}

bootstrap();
```

---

### 二、 核心步骤深度分析

#### 1. 为什么使用 `async/await` 和 `bootstrap` 函数？
*   **异步初始化**：现代架构中，启动前通常需要处理异步逻辑（如连接数据库、读取远程配置中心）。
*   **作用域封装**：将启动逻辑封装在函数内，避免全局变量污染，并方便进行单元测试或集成测试时的程序调用。

#### 2. `NestFactory.create(AppModule)` 背后发生了什么？
这是 NestJS 最繁重的工作阶段。当你调用这个方法时，框架内部执行了以下逻辑：

*   **实例化 IoC 容器**：创建一个内部的 `NestContainer`，这是存放所有单例对象（Provider）的仓库。
*   **元数据扫描（Metadata Scanner）**：
    *   读取 `AppModule` 装饰器里的 `imports`, `controllers`, `providers`。
    *   递归进入每个子模块，构建出一棵 **依赖关系树（Dependency Graph）**。
*   **依赖解析与实例化**：
    *   按照拓扑排序算法，先实例化没有依赖的 Provider，再实例化依赖它们的 Provider。
    *   利用我们在第一阶段提到的 `Reflect Metadata` 读取构造函数参数类型，完成自动注入。
*   **适配器初始化**：
    *   默认选择 `ExpressAdapter`。如果你配置了 Fastify，此时会初始化底层 Fastify 引擎。

#### 3. `app.listen(3000)` 的本质
*   NestJS 并不直接处理 TCP 数据包，它是一个**抽象层**。
*   此行代码会通知底层的 HTTP 平台（Express/Fastify）调用其原生的 `.listen()` 方法。
*   在监听成功前，Nest 会触发所有模块的 **生命周期钩子**（如 `onModuleInit`）。

---

### 三、 平台切换：从 Express 到 Fastify

架构师必须具备根据业务场景切换底层的能力。在 `main.ts` 中可以指定底层引擎：

```typescript
import { NestFactory } from '@nestjs/core';
import { FastifyAdapter, NestFastifyApplication } from '@nestjs/platform-fastify';
import { AppModule } from './app.module';

async function bootstrap() {
  // 明确指定泛型以获得更好的类型提示
  const app = await NestFactory.create<NestFastifyApplication>(
    AppModule,
    new FastifyAdapter()
  );
  await app.listen(3000, '0.0.0.0'); // Fastify 默认监听 127.0.0.1，外部访问需指定 0.0.0.0
}
```

---

### 四、 面试题储备

**Q1：NestJS 应用启动时，是先初始化 Controller 还是先初始化 Provider？**
*   **A**：先初始化 Provider。因为 Controller 依赖 Provider（通常通过构造函数注入），按照 IoC 容器的逻辑，必须先准备好所有的依赖项，才能成功实例化 Controller。

**Q2：如果两个 Module 循环引用，`main.ts` 启动会发生什么？**
*   **A**：如果直接在 `imports` 中循环引用且没有处理，NestJS 会因为无法确定实例化顺序而抛出错误。通常需要使用 `forwardRef()` 来破解循环依赖，或者从架构层面重构。

**Q3：`NestFactory.create` 方法返回的 `app` 对象是什么类型？**
*   **A**：默认是 `INestApplication` 接口实例。它采用**外观模式（Facade Pattern）**，将复杂的底层逻辑封装起来，提供如 `useGlobalPipes`、`setGlobalPrefix` 等顶层 API。

---

### 五、 实际开发技巧

在 `main.ts` 中，我们通常还会做以下三件事（后续章节详讲）：
1.  **设置全局前缀**：`app.setGlobalPrefix('api/v1')`。
2.  **开启 CORS**：`app.enableCors()` 处理跨域。
3.  **集成 Swagger**：在启动前配置文档。

**任务：**
打开你刚创建的项目中的 `main.ts`，尝试添加 `app.setGlobalPrefix('v1')`，并思考如果我想在启动时打印当前监听的端口，应该在哪个环节获取？

## 第三讲：NestFactory 源码分析

在 `main.ts` 中，我们调用的 `NestFactory` 是整个框架的“点火装置”。理解它的源码逻辑，能帮你真正掌握 NestJS 是如何从零构建起复杂的依赖关系的。

---

### 一、 NestFactory 的本质

`NestFactory` 并不是一个普通类，它是一个 **`NestFactoryStatic`** 类的实例。

其源码核心定义（简化版）：
```typescript
export class NestFactoryStatic {
  // 核心方法：创建 Nest 应用
  public async create<T extends INestApplication = INestApplication>(
    module: any,
    serverOrOptions?: any,
    options?: NestApplicationOptions,
  ): Promise<T> {
    // 1. 初始化底层 Web 适配器 (Express 或 Fastify)
    const [httpServer, appOptions] = this.createHttpServer(serverOrOptions);
    
    // 2. 创建 IoC 容器 (NestContainer)
    const container = new NestContainer(appOptions);
    
    // 3. 核心初始化流程
    await this.initialize(module, container, appOptions, httpServer);
    
    // 4. 返回一个封装后的 NestApplication 实例
    return this.createNestInstance<T>(new NestApplication(container, httpServer, appOptions));
  }
}
```

---

### 二、 源码核心链路（四大步骤）

当你执行 `NestFactory.create(AppModule)` 时，内部依次发生了以下动作：

#### 1. 实例化 NestContainer（IoC 容器）
这是 Nest 的“仓库”。所有的 Module、Provider、Controller 实例最终都会存在这个 `container` 对象里。

#### 2. 启动 DependenciesScanner（依赖扫描器）
这是最关键的一步。
*   它会从 `AppModule` 开始，递归读取 `@Module()` 装饰器中的 `imports`。
*   通过 **MetadataScanner** 扫描所有类上的装饰器，识别出谁是 Controller，谁是 Provider。
*   构建依赖图谱。

#### 3. 运行 InstanceLoader（实例加载器）
扫描完所有的“设计图”后，`InstanceLoader` 开始干活：
*   它通过**拓扑排序**算法决定实例化顺序（先实例化被依赖的，再实例化主动依赖的）。
*   利用 `Reflect.getMetadata` 获取构造函数参数。
*   调用 `new Provider(...)` 并存入容器。

#### 4. 适配器绑定（Adapter）
Nest 并不直接操作 HTTP。在 `initialize` 过程中，它会将路由映射逻辑交给底层适配器（如 `ExpressAdapter`），并最终创建原生的 Node.js HTTP Server。

---

### 三、 两个高频使用的变体方法

除了 `create()`，架构师必须知道另外两个源码暴露的方法：

1.  **`createMicroservice()`**：
    *   **区别**：它不会启动 HTTP 服务器，而是初始化指定的传输层（如 Redis, MQTT, gRPC）。
2.  **`createApplicationContext()`**：
    *   **架构用途**：创建一个“纯净”的 IoC 容器，不包含任何 Web/微服务功能。常用于**脚本任务、定时任务 (Cron Jobs) 或 CLI 工具**。它只负责依赖注入，不监听端口。

---

### 四、 面试深度：如何回答“Nest 启动原理”？

**面试官：请结合源码聊聊 NestJS 的启动过程。**

**核心回答点：**
1.  **入口**：通过 `NestFactoryStatic` 创建 `NestContainer`。
2.  **扫描**：`DependenciesScanner` 递归解析模块树，利用 `Reflect Metadata` 提取类元数据。
3.  **依赖解析**：`InstanceLoader` 负责生产实例。它通过递归或循环依赖检测，确保所有 Provider 在 Controller 实例化之前已经准备就绪。
4.  **抽象层**：NestJS 采用了**适配器模式**，通过 `HttpAdapterHost` 将框架逻辑与具体的底层平台（Express/Fastify）解耦。

---

### 五、 实际开发启发

*   **性能优化**：如果你只需要跑一个清理数据库的脚本，不要用 `NestFactory.create`，因为它会拉起整个 Web 环境。请使用 `NestFactory.createApplicationContext`，这能显著降低内存占用和启动时间。
*   **调试技巧**：如果启动报错 `Undefined dependency`，通常是因为 `DependenciesScanner` 在扫描阶段发现某个参数的元数据丢失（通常是循环引用或没加 `@Injectable()`）。

## 第四讲：Module 扫描机制

在上一讲中，我们提到了 `DependenciesScanner`。这一讲我们深入探讨 NestJS 是如何从一个根模块（AppModule）出发，递归“爬行”出整个应用架构图的。

---

### 一、 扫描的核心流程

Nest 的模块扫描是一个**递归过程**。其基本逻辑如下：

1.  **起点**：`NestFactory` 接收到 `AppModule`。
2.  **提取元数据**：通过 `Reflect.getMetadata` 读取该类上 `@Module()` 装饰器里的 `imports`、`controllers`、`providers` 和 `exports`。
3.  **递归进入**：对 `imports` 数组中的每个模块，重复步骤 2。
4.  **去重与存储**：内部使用一个 `Map<string, Module>` 结构存储。键名（Key）通常是模块类的名称。如果扫描过程中发现模块已存在，则跳过（避免无限循环，除非是动态模块）。

---

### 二、 模块定义的四个核心属性（底层视角）

在扫描过程中，Nest 对这四个数组的处理逻辑完全不同：

*   **`imports`**：建立模块间的**拓扑关系**。被引入的模块会先于当前模块完成扫描。
*   **`providers`**：被存入当前模块的 **私有存储区**。默认情况下，Provider 的作用域仅限于声明它的模块。
*   **`controllers`**：绑定到当前模块。扫描器会立即解析其构造函数，准备建立路由映射。
*   **`exports`**：这是**授权书**。只有列在 `exports` 里的 Provider，才能被其他引入此模块的模块所使用。

---

### 三、 Global Module（全局模块）的特殊处理

在扫描时，如果发现某个模块标记了 `@Global()` 装饰器：

1.  **全局注册**：扫描器会将该模块标记为 `isGlobal: true`。
2.  **自动注入**：当其他模块解析依赖时，Nest 不仅会在当前模块的 `imports` 中找，还会去这个“全局池”里找。
3.  **架构警告**：全局模块虽然方便，但会破坏模块的封装性，导致依赖关系不透明。在企业级架构中，通常只用于 **Config**、**Database**、**Logger** 等基础库。

---

### 四、 源码关键类：`ModuleCompiler`

在扫描期间，`ModuleCompiler` 负责将你写的类转换成 Nest 内部的 `Module` 实例。

*   **静态模块**：直接处理类引用。
*   **动态模块 (Dynamic Module)**：如果发现 `imports` 里是一个对象（如 `TypeOrmModule.forRoot()`），编译器会执行该方法，获取返回的 `module`、`providers` 等，然后再进行扫描。

---

### 五、 面试题储备

**Q1：为什么在 A 模块定义了 Provider，在 B 模块引入了 A 模块，B 模块的 Controller 还是无法使用该 Provider？**
*   **A**：因为没有在 A 模块的 `exports` 数组中导出该 Provider。Nest 的模块是强封装的，`imports` 只是引入了模块的“访问权”，具体的 Provider 必须显式导出。

**Q2：Nest 如何处理循环依赖（Circular Dependency）？**
*   **A**：
    1.  如果在扫描阶段发现模块 A 引入 B，B 又引入 A，Nest 会因无法确定顺序而报错。
    2.  解决方案：使用 `forwardRef(() => BModule)`。这会在扫描时告诉 Nest：“先给个占位符，等两边都扫完了再把引用关联上”。

**Q3：模块扫描是在什么时候发生的？**
*   **A**：在 `NestFactory.create()` 异步方法执行期间，`app.listen()` 执行之前。如果扫描失败，应用会直接抛出异常并停止启动。

---

### 六、 实际开发建议：模块拓扑可视化

在大型架构中，模块多了容易乱。
*   **技巧**：可以使用 Nest 官方提供的 **`Nest Tools`** 或第三方包 `nestjs-viz`。
*   **原理**：它们其实就是通过源码暴露的 `app.get(ModulesContainer)` 获取扫描后的 Map 对象，并将其转化为图形化展示。

## 第五讲：Metadata 收集机制

NestJS 的灵魂在于**装饰器（Decorators）**，而装饰器的核心是 **元数据（Metadata）**。它是 NestJS 实现自动化路由映射、依赖注入、权限控制等功能的基础。

---

### 一、 核心基础：reflect-metadata

NestJS 强依赖 `reflect-metadata` 这个库。它在内存中维护了一张“哈希表”，可以将任意数据挂载在“类”、“方法”或“属性”上，而不会破坏对象本身的结构。

**底层 API：**
*   `Reflect.defineMetadata(key, value, target, propertyKey)`: 存储元数据。
*   `Reflect.getMetadata(key, target, propertyKey)`: 获取元数据。

---

### 二、 Nest 如何利用 Metadata 标记“身份”

当你写下 `@Controller('/user')` 时，Nest 源码内部其实做了类似以下的操作：

```typescript
// 伪代码实现 @Controller
function Controller(prefix: string): ClassDecorator {
  return (target) => {
    // 将 'path' 元数据标记在类上
    Reflect.defineMetadata('__path__', prefix, target);
    // 标记这个类是一个 Controller 角色
    Reflect.defineMetadata('__scope__', 'CONTROLLER', target);
  };
}
```

在启动扫描阶段（上一讲内容），`DependenciesScanner` 会遍历所有的类，通过 `Reflect.getMetadata('__scope__', class)` 判断：
*   如果是 `CONTROLLER` -> 注册路由。
*   如果是 `INJECTABLE` -> 存入 IoC 容器准备注入。

---

### 三、 通用装饰器：@SetMetadata()

在实际开发中，我们经常需要自定义元数据（例如：标记某个接口需要的权限）。Nest 提供了官方工具：

**1. 定义：**
```typescript
@SetMetadata('roles', ['admin'])
@Get('profile')
getProfile() {}
```

**2. 架构优化（推荐）：**
直接使用 `SetMetadata` 会导致硬编码字符串。企业级写法是封装自定义装饰器：
```typescript
export const Roles = (...roles: string[]) => SetMetadata('roles', roles);

// 使用时
@Roles('admin', 'user')
@Get()
findAll() {}
```

---

### 四、 运行时提取：Reflector 类

Metadata 收集后的目的是为了**使用**。在 Guard（守卫）或 Interceptor（拦截器）中，我们使用 `Reflector` 提取这些信息。

**核心代码示例：**
```typescript
@Injectable()
export class RolesGuard implements CanActivate {
  constructor(private reflector: Reflector) {}

  canActivate(context: ExecutionContext): boolean {
    // 从当前处理的方法上获取 'roles' 元数据
    const roles = this.reflector.get<string[]>('roles', context.getHandler());
    // ...逻辑判断
    return true;
  }
}
```

---

### 五、 面试题储备

**Q1：TypeScript 的装饰器和元数据在编译后会消失吗？**
*   **A**：装饰器本身会编译成高阶函数。元数据如果开启了 `emitDecoratorMetadata`，会被编译成 `__metadata` 函数调用。只要在运行时引入了 `reflect-metadata` 库，这些信息就会保存在内存中，不会消失。

**Q2：如何获取一个类构造函数参数的类型信息？**
*   **A**：使用 `Reflect.getMetadata('design:paramtypes', target)`。这是 TS 编译器预定义的三个标准元数据 Key 之一（另外两个是 `design:type` 和 `design:returntype`）。Nest 正是靠这个实现自动依赖注入的。

**Q3：`Reflector.getAllAndOverride` 和 `getAllAndMerge` 有什么区别？**
*   **A**：
    *   `getAllAndOverride`: 优先级覆盖。通常用于“方法”上的装饰器覆盖“类”上的。例如：类标记了全接口需要 Admin 权限，但某个方法标记了 Public，则覆盖类配置。
    *   `getAllAndMerge`: 合并。将类和方法上的元数据合并成一个数组。

### 六、 实际开发启发

*   **解耦逻辑**：如果你想实现一个全局日志功能，但只想记录部分接口。不要在 Controller 里写 `if`，而是定义一个 `@Log()` 装饰器挂载 Metadata，然后在 `Interceptor` 中根据是否存在该 Metadata 来决定是否记录。
*   **AOP 思想**：Metadata 机制是 Nest 实现 AOP（面向切面编程）的基石。它让业务逻辑（Controller）和横切逻辑（权限、日志、缓存）通过“注解”彻底解耦。
