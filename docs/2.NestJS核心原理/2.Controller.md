## 第六讲：Controller 原理

在 NestJS 中，**Controller（控制器）** 是应用程序的“大门”。它的核心职责是接收传入的 HTTP 请求，并将这些请求分发到应用程序的其他部分（如 Service）进行处理，最后将结果返回给客户端。

本讲我们重点剖析 **Controller 的底层工作原理及其在 IoC 容器中的特殊地位**。

---

### 一、 Controller 的核心定义与职责

在传统的 MVC（Model-View-Controller）或者现代的 MVVM、三层架构中，Controller 都是负责**控制流**的部件。

在 NestJS 中，一个标准的 Controller 定义如下：

```typescript
import { Controller, Get } from '@nestjs/common';
import { ArticleService } from './article.service';

@Controller('articles') // 1. 路由前缀装饰器
export class ArticleController {
  // 2. 构造函数：声明式依赖注入
  constructor(private readonly articleService: ArticleService) {}

  @Get() // 3. HTTP 方法与子路由装饰器
  findAll() {
    return this.articleService.findAll();
  }
}
```

#### 职责边界（Thin Controller 思想）
在企业级架构中，Controller 应当保持**极度轻量（Thin Controller）**：
1. **只做分发与搬运**：负责接收请求参数（Query, Body, Param 等），并调用 Service 层的具体业务逻辑。
2. **不含业务逻辑**：不应该在 Controller 中直接编写 SQL 语句、复杂的算法或直接操作底层数据库。
3. **统一响应格式**：负责最终 HTTP 响应状态码及基础数据结构的返回。

---

### 二、 Controller 的工作原理与 IoC

我们在第三讲和第四讲中学习了 `NestFactory` 和 `Module` 的扫描机制。那么，IoC 容器是如何处理 Controller 的呢？

#### 1. Controller 在 Module 中的特殊注册
在 `@Module()` 装饰器中，有专门的 `controllers` 数组：

```typescript
@Module({
  controllers: [ArticleController], // 注册到控制器数组中
  providers: [ArticleService],
})
export class ArticleModule {}
```

* **不同于 Provider**：在 Nest 内部，`providers` 和 `controllers` 的解析逻辑不同。`controllers` 中的类是**路由挂载的目标**，它们不能被 `export`（导出），也不能被其他模块 `import` 注入。你不能在其他类（比如另一个 Service）的构造函数中注入一个 Controller。
* **依赖的消费者**：Controller 本身是**依赖的消费者**。当 `InstanceLoader`（实例加载器）开始干活时，它发现 `ArticleController` 的构造函数需要一个 `ArticleService` 实例。IoC 容器会在当前的模块上下文中寻找 `ArticleService` 的单例对象并注入进去。

#### 2. 实例化过程（拓扑依赖解析）
1. `DependenciesScanner` 扫描到 `ArticleModule`，发现其下有一个 `ArticleController`。
2. 扫描器读取该 Controller 构造函数的参数元数据（利用反射获取 `ArticleService` 的类型信息）。
3. 框架首先去实例化 `ArticleService`，并将其缓存到 IoC 容器中。
4. 容器调用 `new ArticleController(articleServiceInstance)` 创建控制器实例。

---

### 三、 核心装饰器 `@Controller` 的底层实现

我们在第五讲提到了元数据收集，实际上 `@Controller()` 装饰器的主要工作就是往类上挂载元数据：

```typescript
// NestJS 源码逻辑简化版
export function Controller(prefixOrOptions?: string | ControllerOptions): ClassDecorator {
  const defaultPath = '/';
  // 解析路由前缀，比如 'articles'
  const path = typeof prefixOrOptions === 'string' ? prefixOrOptions : prefixOrOptions?.path || defaultPath;

  return (target: object) => {
    // 1. 标记它是 CONTROLLER 类型，供 DependenciesScanner 识别
    Reflect.defineMetadata(PATH_METADATA, path, target);
    // 2. 挂载可选的宿主、作用域等配置
    if (typeof prefixOrOptions === 'object') {
      Reflect.defineMetadata(SCOPE_METADATA, prefixOrOptions.scope, target);
      Reflect.defineMetadata(HOST_METADATA, prefixOrOptions.host, target);
    }
  };
}
```

当 `main.ts` 启动、适配器绑定路由时，底层的 `RouterExplorer` 会提取类上的 `PATH_METADATA`（如 `/articles`），再配合方法上的元数据（如 `@Get`）合并出完整的 URL，然后注册到底层的 Express 或 Fastify 路由表中。

### 四、 面试题储备

**Q1：为什么 Controller 不需要标记 `@Injectable()` 装饰器？**

* **A**：
  1. `@Controller()` 装饰器内部已经实现了类似的机制。它在类上挂载了特定的元数据，可以让 Nest 知道它是一个由 IoC 容器管理并需要解析依赖的类。
  2. 职责不同。`@Injectable()` 声明的类是作为被注入的 **Provider（服务提供者）**；而 Controller 是**请求的终点与分发者**，它不会作为依赖项被注入到其他类中。因此，用专属的 `@Controller()` 可以保证架构角色划分的清晰性。

**Q2：如果两个 Controller 拥有相同的路由前缀（比如都配置了 `@Controller('user')`），启动时会报错吗？**

* **A**：不会报错。在路由扫描阶段，Nest 会依次将各 Controller 下的方法注册到底层适配器上。如果方法上的路径也一样（例如两个 Controller 下都有 `GET /user`），此时不会抛出编译或启动错误，但是底层 Web 框架（如 Express）在处理请求时会**采用“先注册先匹配”的原则**。这在大型项目里属于隐蔽的 bug，因此应当在架构上避免路由冲突。

**Q3：Controller 默认是单例（Singleton）的吗？如果它注入了一个请求作用域（Request Scope）的 Service，会发生什么？**

* **A**：默认是单例的。但是，如果一个 Controller 注入了一个生命周期为 Request 作用域的 Service，那么这个 Controller 本身的生命周期也会被自动**“传染”提升为 Request 作用域**。这意味着每次有新的 HTTP 请求进来时，Nest 都会为该请求创建一个新的 Controller 实例，并在请求结束后销毁它。这会增加垃圾回收（GC）的开销，开发中如非必要，应尽量保持默认的单例模式。

## 第七讲：路由映射机制

在上一讲中，我们学习了 Controller 的基本原理。这一讲我们深入探讨 **NestJS 的路由映射机制**。当一个 HTTP 请求到达服务器时，NestJS 是如何精确定位到某一个 Controller 的具体方法（Handler）的？

---

### 一、 路由的基础拼接规则

NestJS 的路由地址是由 **类（Controller）的路径前缀** 与 **方法（Handler）的路径后缀** 拼接而成的。

```typescript
@Controller('articles') // 1. 类级别前缀
export class ArticleController {
  
  @Get() // 2. 方法级别：默认为空
  findAll() {} // 最终匹配：GET /articles

  @Get(':id') // 3. 方法级别：带参数
  findOne() {} // 最终匹配：GET /articles/:id

  @Post('draft') // 4. 方法级别：子路径
  createDraft() {} // 最终匹配：POST /articles/draft
}
```

#### 核心要点
* **斜杠（`/`）的自动处理**：无论你在 `@Controller('articles')` 还是 `@Get('info')` 中是否写了前置或后置斜杠，NestJS 都会在编译时自动对其进行规范化处理（例如：`articles` + `/info` 和 `/articles` + `info` 最终都会被解析为 `/articles/info`）。
* **HTTP 请求方法映射**：Nest 提供了 `@Get()`, `@Post()`, `@Put()`, `@Delete()`, `@Patch()`, `@Options()`, `@Head()`, `@All()` 等修饰方法的装饰器。它们在底层都继承自 `@RequestMapping()` 装饰器。

---

### 二、 高级路由匹配与通配符

为了满足复杂的业务场景，Nest 提供了灵活的路由匹配机制。

#### 1. 动态路由（Path Parameters）
使用冒号 `:` 来定义路由参数，常用于标识特定资源：
```typescript
@Get(':id')
findOne() {} // 匹配 GET /articles/123, GET /articles/abc
```

#### 2. 通配符路由（Wildcards）
NestJS 支持在路由中使用星号 `*` 作为通配符：
```typescript
@Get('ab*cd')
findAll() {} // 匹配 GET /articles/abcd, GET /articles/ab_cd, GET /articles/abecd 等
```
* **注意**：字符 `?`、`+`、`*` 和 `()` 可以在路由路径中使用，它们是其对应正则表达式子集的映射。

---

### 三、 底层路由挂载原理

Nest 本身并不直接负责网络底层的路由寻址，而是通过 **适配器模式（Adapter Pattern）** 委托给底层框架（如 Express 或 Fastify）。

#### 路由注册的核心链路
1. **启动扫描**：在 `NestFactory.create` 阶段，`RouterExplorer` 类会扫描所有被 IoC 容器实例化的 Controller。
2. **提取元数据**：
   * 通过 `Reflect.getMetadata` 提取类上的路径前缀。
   * 遍历该类的所有原型方法，通过 `@Get`、`@Post` 等装饰器提取 **请求方法（Method）** 和 **子路径（Path）**。
3. **委托挂载**：
   * 框架内部将“前缀”与“子路径”进行拼接，得到完整的路由字符串。
   * 接着，调用底层适配器的路由注册方法。
   * *如果是 Express*：相当于在后台调用 `expressApp.get('/articles/:id', handler)` [1]。
   * *如果是 Fastify*：相当于调用 `fastifyApp.route({ method: 'GET', url: '/articles/:id', handler })`。

---

### 四、 面试题储备

**Q1：在路由映射中，如果有两个路由存在交集（例如：`GET /articles/:id` 和 `GET /articles/draft`），哪个会被优先匹配？**

* **A**：
  在同一个 Controller 中，路由的匹配顺序遵循 **“先定义先匹配”** 的原则。
  如果你的代码写成：
  ```typescript
  @Get(':id')
  findOne() {}
  
  @Get('draft')
  findDraft() {}
  ```
  当客户端访问 `GET /articles/draft` 时，请求会被 `findOne()` 拦截，并将 `'draft'` 作为参数 `id` 进行处理。
  **最佳实践**：必须把**具体的、静态的路由**（如 `draft`）写在**动态的、通配的路由**（如 `:id`）之前。

**Q2：NestJS 是否支持正则路由？如果支持，它是如何处理的？**

* **A**：
  支持。因为默认底层是 Express，其路由解析使用的是 `path-to-regexp` 库。你可以直接在路由装饰器中传入正则表达式字符串。不过在企业级开发中，为了代码的可读性和跨平台兼容性（如果切换到 Fastify），建议尽量避免过于复杂的正则路由，转而在 Controller 内部使用业务逻辑或管道（Pipe）进行格式校验。

**Q3：如果我们将底层从 Express 切换到 Fastify，通配符路由（`*`）会受到影响吗？**

* **A**：
  会受到影响。Express 的路由解析相对宽松，支持类似 `ab*cd` 的中间通配。而 Fastify 采用的是基于基数树（Radix Tree）的路由，它不支持在路径中间使用 `*` 这种非标准通配符 [1]；Fastify 的通配符通常只能放在路径的末尾（如 `/*`）[1]。因此，编写通配符路由时需要注意底层平台的差异。

## 第八讲：参数装饰器

在上一讲中，我们完成了路由的映射。当请求成功路由到对应的 Handler（处理器方法）时，我们通常需要获取客户端传递过来的数据。

NestJS 提供了丰富的**参数装饰器（Parameter Decorators）**，用于快捷地从底层的 HTTP 请求对象中提取所需的参数。这一讲我们将深入学习并掌握这些内置参数装饰器的底层工作原理。

---

### 一、 常用内置参数装饰器对照表

NestJS 的参数装饰器是对底层 Web 框架（如 Express/Fastify）原生属性的封装。下面是高频使用的内置装饰器与原生的对应关系：

| NestJS 参数装饰器            | 对应的底层 Express 对象属性         | 提取的数据内容                          |
| :--------------------------- | :---------------------------------- | :-------------------------------------- |
| **`@Request()` / `@Req()`**  | `req`                               | 完整的 HTTP 请求对象                    |
| **`@Response()` / `@Res()`** | `res`                               | 底层 HTTP 响应对象                      |
| **`@Param(key?)`**           | `req.params` / `req.params[key]`    | 路径参数（动态路由参数）                |
| **`@Query(key?)`**           | `req.query` / `req.query[key]`      | 查询参数（URL中 `?` 后面的键值对）      |
| **`@Body(key?)`**            | `req.body` / `req.body[key]`        | 请求体（POST/PUT 传递的数据）           |
| **`@Headers(name?)`**        | `req.headers` / `req.headers[name]` | HTTP 请求头                             |
| **`@Ip()`**                  | `req.ip`                            | 客户端的 IP 地址                        |
| **`@Session()`**             | `req.session`                       | Session 对象（需要引入 Session 中间件） |

---

### 二、 常见参数获取方式详解

在开发博客系统时，我们会频繁使用 `@Param`、`@Query` 和 `@Body`。

#### 1. 路径参数 `@Param()`
用于获取动态路由中的参数。有两种提取方式：
```typescript
// 方式 A：获取整个 params 对象
@Get(':id')
findOne(@Param() params: any) {
  console.log(params.id); // 输出获取到的 id
}

// 方式 B：直接提取指定 key（推荐，代码更清晰）
@Get(':id')
findOne(@Param('id') id: string) {
  console.log(id);
}
```

#### 2. 查询参数 `@Query()`
用于分页、筛选或搜索。例如在博客中获取文章列表：
```typescript
// 访问路径示例: /articles?page=1&limit=10
@Get()
findAll(@Query('page') page: string, @Query('limit') limit: string) {
  return `正在获取第 ${page} 页的文章，每页展示 ${limit} 条`;
}
```

#### 3. 请求体 `@Body()`
用于接收客户端提交的 JSON 或 Form 表单数据，常用于创建或修改资源：
```typescript
@Post()
create(@Body() createArticleDto: any) {
  // createArticleDto 包含了客户端发送过来的完整 Body 数据
  return createArticleDto;
}
```

---

### 三、 `@Res()` / `@Response()` 的“底层陷阱”

在 NestJS 中，**不推荐**轻易注入 `@Res()`。

#### 1. 什么是库特有模式（Library-specific mode）？
一旦在 Handler 的参数中注入了 `@Res()` 或 `@Response()`，NestJS 会自动切换到**库特有模式**：
* 此时，NestJS **不再负责**该 Handler 的自动响应（即你直接 `return` 任何对象，客户端都收不到，请求会一直挂起直到超时）。
* 你必须手动调用底层响应对象的方法（如 `res.send()` 或 `res.json()`）来结束响应。

```typescript
// 错误示范：会导致请求挂起
@Get()
findAll(@Res() res) {
  return 'hello'; // 客户端永远收不到 'hello'
}

// 库特有模式下的正确写法
@Get()
findAll(@Res() res) {
  res.status(200).send('hello');
}
```

#### 2. 完美的折中方案：`passthrough: true`
如果你只需要通过 `res` 来进行某些底层操作（如：写入 Cookie、设置特殊的 Header），但依然希望由 NestJS 来接管返回值和生命周期，可以配置 `passthrough` 选项：

```typescript
@Get()
findAll(@Res({ passthrough: true }) res) {
  res.cookie('test-cookie', '123456'); // 手动操作底层的 response 对象
  return 'hello'; // 依然可以由 Nest 自动包装并返回
}
```

### 四、 面试题储备

**Q1：`@Param()` 和 `@Query()` 在场景应用上有什么区别和规范？**

* **A**：
  * **`@Param()`（路径参数）**：用于**精确定位特定资源**。在 RESTful 架构中，它是资源路径的一部分，代表资源的唯一标识。例如：`/articles/10`，这里的 `10` 是一个具体资源的主键。
  * **`@Query()`（查询参数）**：用于**过滤、排序、分页或搜索资源**。它不决定资源的唯一性，而是决定展示资源的方式。例如：`/articles?status=published&sort=desc`。

**Q2：如果我直接在 Controller 方法里声明了参数类型（例如 `id: number`），NestJS 会自动将字符串类型的路由参数转化为数字吗？**

* **A**：
  默认不会。通过 HTTP 传输的所有路径参数、查询参数本质上都是**字符串**（String）。即使你在 TS 中写了 `@Param('id') id: number`，在运行时 `id` 的类型仍然是 `string`。
  要实现自动类型转换，通常需要借助 NestJS 的 **Pipe（管道）**，如 `@Param('id', ParseIntPipe)`，或者全局启用 `ValidationPipe` 并配置 `transform: true`。

**Q3：为什么不推荐在 Controller 中直接使用 `@Req() req` 和 `@Res() res` 来编写业务代码？**

* **A**：
  1. **失去框架抽象优势**：直接操作原生 `req` 和 `res` 会使代码与底层平台（如 Express）强耦合。一旦将来需要切换到高性能的 Fastify 平台，大量平台特有的 API 就会失效。
  2. **破坏 AOP 横切面**：绕过 Nest 自身的响应流，会导致全局的拦截器（Interceptor）、序列化工具等失效，增加测试和维护的成本。

## 第九讲：自定义参数装饰器

在前一讲中，我们学习了 NestJS 的内置参数装饰器（如 `@Param`、`@Query` 和 `@Body`）。它们能帮我们提取标准的 HTTP 请求属性。

但在实际开发中，尤其是在构建博客系统的身份认证、权限管理时，我们经常需要重复执行某些参数提取逻辑（例如：从 `req.user` 中提取当前登录的用户信息）。为了避免代码冗余、提升代码的规范性与可读性，NestJS 允许我们创建**自定义参数装饰器（Custom Parameter Decorators）**。

---

### 一、 为什么需要自定义参数装饰器？

在博客系统的后台，当博主发表文章时，系统需要记录“是谁发表了这篇文章”。在鉴权守卫（Auth Guard）通过后，用户信息通常会被挂载在 `req.user` 对象上。

**如果不使用自定义装饰器，我们的代码可能会这样写：**

```typescript
@Post()
create(@Req() req, @Body() createArticleDto: any) {
  const user = req.user; // 每次都需要从 req 对象中手动提取
  return this.articleService.create(user.id, createArticleDto);
}
```
这种写法存在两个弊端：
1. **显式依赖了整个 `@Req()` 对象**：降低了单元测试的便利性，因为在测试时你需要去模拟整个复杂的 Request 对象。
2. **代码重复**：任何需要获取当前用户信息的接口，都需要重复写一遍 `req.user`。

为了解决这个问题，我们可以自定义一个 `@User()` 装饰器。

---

### 二、 如何创建自定义参数装饰器

NestJS 提供了 `createParamDecorator` 辅助函数来帮助我们定义自定义参数装饰器。

#### 1. 定义装饰器：`user.decorator.ts`

```typescript
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const User = createParamDecorator(
  (data: string, ctx: ExecutionContext) => {
    // 1. 获取底层的 HTTP 请求对象
    const request = ctx.switchToHttp().getRequest();
    const user = request.user; // 这里的 user 通常由鉴权守卫 (Guard) 在此前注入

    // 2. 如果装饰器传入了特定的 key，则返回对应属性；否则返回整个 user 对象
    return data ? user?.[data] : user;
  },
);
```

#### 2. 在 Controller 中使用装饰器

```typescript
@Post()
create(
  @User() user: any, // 获取完整的用户对象
  @User('id') userId: string, // 只提取用户 ID 属性
  @Body() createArticleDto: any,
) {
  console.log('当前登录用户ID:', userId);
  return { userId, ...createArticleDto };
}
```

---

### 三、 深入 ExecutionContext（执行上下文）

在定义自定义装饰器时，我们接触到了一个非常关键的参数：**`ExecutionContext`（执行上下文）**。

#### 1. 为什么不是 `req`？
NestJS 是一个**跨平台、跨协议**的框架。它不仅可以运行在 HTTP 环境（Express/Fastify）中，还可以支持 WebSockets、微服务（Microservices）、gRPC 等。

为了让你的装饰器、守卫、拦截器能够在不同的协议中复用，NestJS 抽象出了 `ExecutionContext`。它继承自 `ArgumentsHost`，提供了一套统一的、与具体协议无关的方法。

#### 2. 核心 API 工作流
在 HTTP 场景下，我们通过以下链式调用来获取原生对象：
```typescript
const request = ctx.switchToHttp().getRequest(); // 切换到 HTTP 上下文，并获取 Request 对象
const response = ctx.switchToHttp().getResponse(); // 获取 Response 对象
```

---

### 四、 面试题储备

**Q1：自定义参数装饰器可以像 Provider 那样，在内部通过依赖注入（DI）引入其他 Service 吗？**

* **A**：
  **不能**。自定义参数装饰器（以及所有 TypeScript 装饰器）是在代码编译和类声明时就已经确定并执行的高阶函数，它们不属于 NestJS 的 IoC 容器管辖。
  由于装饰器内部无法利用 DI 注入服务，如果你需要在获取参数时执行复杂的异步查库操作，推荐的方案是：**利用 Guard（守卫）或 Interceptor（拦截器）在请求到达 Controller 之前将数据查出并挂载到 `req` 上，或者在控制器参数中使用 Pipe（管道）来处理。**

**Q2：`createParamDecorator` 中的 `data` 参数有什么用？它能接收哪些类型的数据？**

* **A**：
  `data` 是客户端在使用该装饰器时，传给装饰器的第一个参数。例如 `@User('email')`，此时 `data` 的值就是 `'email'`。它可以是任何类型（如 `string`、`string[]`、甚至是配置对象 `object`），取决于你在装饰器逻辑中如何解析它。

**Q3：自定义参数装饰器可以与 NestJS 内置的 Pipe（管道）配合使用吗？**

* **A**：
  可以。NestJS 的参数管道适用于所有参数装饰器（包括自定义的）。
  例如，你可以将提取出的用户 ID 传给一个 `ParseIntPipe` 进行数字转化：
  ```typescript
  @Get('profile')
  getProfile(@User('id', ParseIntPipe) userId: number) {
    return userId;
  }
  ```

## 第十讲：RESTful 设计规范

在前面九讲中，我们已经打通了 NestJS 路由、参数提取、以及自定义处理的整条链路。作为一个包含前后台的博客系统，前后端分离是标准架构，而前后端通信最通用的语言就是 **RESTful API** 规范。

这一讲我们系统学习 **RESTful 架构的设计原则**，以及 **NestJS 框架是如何天然契合并高效支持该规范的**。

---

### 一、 RESTful 核心设计原则

REST（Representational State Transfer，表现层状态转移）是一种软件架构风格，而不是标准。在 RESTful 架构中，我们需要遵循以下核心原则：

1.  **面向资源（Resource-Oriented）**：网络上的所有事物都被抽象为“资源”。
2.  **使用名词，复数优先**：资源的路径（URI）应该只包含名词，且推荐使用复数形式。
    *   *反例（RPC风格）*：`/getArticles`, `/create_article`, `/deleteArticle?id=1`
    *   *正例（RESTful风格）*：`/articles`, `/articles/1`
3.  **使用 HTTP 方法（Verbs）表达动作**：通过 HTTP 的动作来指定对资源执行的操作，而不是在 URL 里写动词。

---

### 二、 HTTP 方法的语义性、安全性与幂等性

设计一个标准的 RESTful API，必须严格遵守 HTTP 方法的语义和机制：

| HTTP 方法    | 对应操作 | 语义描述                   | 安全性 (Safe) | 幂等性 (Idempotent) | NestJS 默认状态码 |
| :----------- | :------- | :------------------------- | :-----------: | :-----------------: | :---------------: |
| **`GET`**    | 获取     | 查询单个或列表资源         |    **是**     |       **是**        |     `200 OK`      |
| **`POST`**   | 新增     | 创建新的资源               |      否       |         否          |   `201 Created`   |
| **`PUT`**    | 更新     | **完整替换**资源的所有字段 |      否       |       **是**        |     `200 OK`      |
| **`PATCH`**  | 更新     | **局部更新**资源的某些字段 |      否       |  否（取决于实现）   |     `200 OK`      |
| **`DELETE`** | 删除     | 销毁或标记删除特定资源     |      否       |       **是**        |     `200 OK`      |

#### 概念辨析：
*   **安全性 (Safe)**：该操作不会改变服务器上的资源状态（只读）。`GET` 是安全的。
*   **幂等性 (Idempotent)**：无论调用一次还是多次，对服务器产生的**副作用（最终状态）**都是一样的。例如 `DELETE /articles/1` 执行一次和执行十次，该文章都是“已被删除”的状态。而 `POST` 每调用一次就会创建一条新数据，因此非幂等。

---

### 三、 HTTP 状态码与 NestJS 的实现

状态码是 RESTful API 的重要反馈机制。NestJS 框架默认根据标准规范映射了状态码：

*   **`200 OK`**：请求成功（通常对应 `GET`, `PUT`, `PATCH`, `DELETE`）。
*   **`201 Created`**：资源创建成功（对应 `POST`）。NestJS 底层会对 `@Post()` 装饰的方法自动返回 `201`。
*   **`204 No Content`**：请求成功但无具体内容返回。例如删除文章后不需要返回任何 JSON。

#### 自定义状态码
如果需要手动更改某个路由的默认返回状态码，可以使用 Nest 提供的 `@HttpCode()` 装饰器：

```typescript
import { Controller, Delete, HttpCode } from '@nestjs/common';

@Controller('articles')
export class ArticleController {
  
  @Delete(':id')
  @HttpCode(204) // 默认是 200，此处显式指定为 204
  remove() {
    // 执行删除逻辑
  }
}
```

---

### 四、 资源嵌套与关系表示

在博客系统中，资源之间存在复杂的关联。例如：**一篇文章（Article）下有多条评论（Comment）**。
RESTful 规范推荐使用**层级嵌套路径**来表示这种父子级关联关系：

```typescript
// 1. 获取某篇文章下的所有评论
// GET /articles/:articleId/comments
@Get(':articleId/comments')
getComments(@Param('articleId') articleId: string) {}

// 2. 在某篇文章下发表一条评论
// POST /articles/:articleId/comments
@Post(':articleId/comments')
createComment(@Param('articleId') articleId: string, @Body() body: any) {}
```

### 五、 面试题储备

**Q1：在资源更新时，PUT 和 PATCH 的区别是什么？我们应该如何选择？**

*   **A**：
    *   **PUT** 是**完全替换（Replace）**。客户端必须提供修改后资源的**所有字段**。如果省略了某字段，服务端通常会用空值或者默认值覆盖原有的值。
    *   **PATCH** 是**局部更新（Modify）**。客户端只需提供需要修改的字段。例如博客系统中，博主只想修改文章标题，使用 `PATCH /articles/1` 传入 `{ "title": "新标题" }` 即可，内容（content）等其他字段保持不变。
    *   **实践选择**：在实际企业开发中，由于完全替换不仅传输成本高，且容易导致未传字段丢失，因此**局部更新（PATCH）的使用频率远高于 PUT**。

**Q2：NestJS 是如何自动化返回类似 400、401、403、404 这些 RESTful 错误状态码的？**

*   **A**：
    NestJS 内部集成了 **Exception Filters（异常过滤器）** 机制。
    框架提供了一系列内置的 HTTP 异常类（如 `BadRequestException`, `UnauthorizedException`, `ForbiddenException`, `NotFoundException`）。
    当我们在代码中抛出这些异常时，NestJS 会自动捕获它们并转换成标准的 RESTful JSON 响应。例如：
    ```typescript
    throw new NotFoundException('该文章不存在');
    ```
    客户端收到的响应会自动被规范化为 `404` 状态码和如下结构：
    ```json
    {
      "statusCode": 404,
      "message": "该文章不存在",
      "error": "Not Found"
    }
    ```

**Q3：什么是 RESTful API 的非状态（Statelessness）原则？在开发中如何体现？**

*   **A**：
    非状态原则要求**每次请求都必须包含理解并完成该请求所需的所有信息**。服务器不应该保存客户端的会话上下文（即不依赖传统服务器端的 Session）。
    在开发中的体现是：用户身份验证不使用 Cookie-Session 机制，而是将 Token 放在 HTTP 请求头（如 `Authorization: Bearer <token>`）中发送。服务器对每次请求都是独立、无状态地进行解析与响应，便于后端服务的横向扩展。

*   `PATCH /admin/articles/:id` - 编辑文章（局部更新）
