# 第1讲：Node.js运行机制

------

## 一、为什么必须学Node.js运行机制？

很多人学NestJS是这样的：

```ts
@Controller()
export class UserController {
  @Get()
  findAll() {
    return this.userService.findAll();
  }
}
```

代码会写。

但是面试官问：

> 为什么 Node.js 单线程却能支持高并发？

不会。

> [!tip]
>
> Node.js 虽然 JavaScript 执行环境是单线程的，但它依靠 **事件驱动（Event Driven）** 和 **非阻塞异步 I/O** 机制实现高并发。
>
> 当遇到网络请求、文件读写、数据库访问等耗时操作时，Node.js 不会阻塞主线程等待结果，而是把这些任务交给底层的 **libuv** 和操作系统处理，主线程继续处理其他请求。
>
> 当异步任务完成后，结果会被放入事件队列，事件循环（Event Loop）再把对应回调取出来执行。因此一个线程就能同时管理大量连接，从而支持高并发。

------

继续问：

> Promise、setTimeout、数据库查询谁先执行？

不会。

> [!tip]
>
>  在 Node.js 中，Promise 的 then 回调属于微任务（MicroTask），会在当前同步代码执行结束后立即执行，因此优先级高于 setTimeout 和数据库查询回调。数据库查询属于 I/O 回调，通常在 Poll 阶段执行；setTimeout 属于 Timers 阶段执行。Promise 一定先执行，而数据库查询和 setTimeout 的先后顺序取决于 I/O 是否已经完成以及 Event Loop 当前所处阶段，因此不能绝对保证谁先执行。

------

继续问：

> 为什么NestJS能同时处理上万个请求？

不会。

> [!tip]
>
> NestJS 能处理上万个并发请求，本质原因不是 NestJS 本身，而是它运行在 Node.js 之上。Node.js 采用单线程 Event Loop 和非阻塞 I/O 模型，当请求涉及数据库、Redis、HTTP 调用等耗时操作时，会交给底层 libuv 和操作系统处理，主线程不会阻塞，可以继续处理其他请求。因此一个线程就能管理大量连接。NestJS只是提供了模块化、依赖注入和装饰器等开发能力，高并发能力主要继承自 Node.js。如果业务中存在大量 CPU 密集型计算，Event Loop 会被阻塞，这时需要使用 Worker Threads、消息队列或者微服务拆分来解决。

------

实际上：

```text

↓

↓
操作系统
```

## 本讲必须掌握的面试回答

如果面试官问：

> Cluster 和 Worker Threads 应该怎么选？

回答：

```text
Cluster用于扩展Web服务吞吐量，充分利用多核CPU处理更多请求；它是多进程模型，每个Worker拥有独立内存和V8实例。

Worker Threads用于处理CPU密集型任务，例如PDF生成、Excel导出、图片压缩、加密和AI推理；它是多线程模型，创建成本更低，并支持共享内存。

通常NestJS线上API服务使用PM2或容器扩容，而具体的CPU重任务会下放到Worker Threads执行。
```

这已经是 Node.js / NestJS 中高级面试关于并发模型的标准答案。
